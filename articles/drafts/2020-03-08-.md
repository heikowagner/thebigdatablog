## 1. Introduction

The development and training of models is fun, but at some point, the exemplary statistician has to ask himself how to keep track of his algorithms and trained models. A central challenge is to ensure that all results can be reproduced easily. Furthermore, further development of the algorithms and models must be possible without much additional effort. During this article we will setup such an environment in the cloud using the Kubernetes technology and and deploy exemplary algorithm and model setup.

## 1.1 Outline

Our setup is based on 3 Containers.

1. A [Gitlab](https://about.gitlab.com/) Container
2. A [Docker Registry](https://docs.docker.com/registry/) Container
3. A [TenserFlow Serving](https://www.tensorflow.org/tfx/guide/serving) Container

Every analyze has to consist of at least 3 necessary files

- Dockerfile
- docker-compose.yaml
- a text file where the code is stored, e.g. stores.r
- a model folder

In the development process the codefile (stores.r) is mounted into the container to make the development process as easy as possible. After the model is fitted the model has to be deployed TensorFlow Serving. The 3 Files have then to be checked into gitlab, here an automatic ci system will check if the code works and works with the model deployed to the TensoFlow Serving using the kubernetes backend. The Dockercontainer will then be build and pushed to the registry. Results can now easily be reproduced with the docker-compose.yaml file by setting an environment variable for the model version and the container version for the algorithm version. A nice feature of our setup that we can control due to the right management at registry and serving who can is allowed to contruct the models and who is allowed to train these models. In a corperate where differnet divisions are in Zuständig for this, this can be important.

## 2.1 Install Helm

Helm is what CRAN is for R or pip for python, a  popular package management system for Kubernetes. To start we need a [local Kubernetes](https://www.thebigdatablog.com/kubernetes-and-docker-running-at-an-orangepi/) Cluster or have access to a cloud cluster and [kubectl installed](https://www.thebigdatablog.com/kubernetes-and-docker-running-at-an-orangepi/) at the client side. Helm has two parts: a client (helm) and a server (tiller). At first we need to install helm at our client with

`Linux:
curl -L https://git.io/get_helm.sh | bash`\
`Windows: choco install kubernetes-helm`

Next we install tiller with `kubectl apply -f https://raw.githubusercontent.com/heikowagner/thebigdatablog/master/kubernetes/tiller.yaml` . To initialize tiller with `helm init --service-account=tiller --history-max 300` . We can now check the tiller deployment with `kubectl get deployment tiller-deploy -n kube-system`.

It’s always a good habbit to create a new namespace for a new project. Therefore we will create a namespace called datascience

```
kubectl create namespace datascience

helm install --namespace datascience --name gitlab --set externalUrl=http://gitlab.varmydomain.com/ stable/gitlab-ce

echo http://$(kubectl get svc --namespace varmygitlab gitlab-gitlab-ce -o jsonpath='{.status.loadBalancer.ingress[0].hostname}') 

helm install --namespace datascience stable/docker-registry
helm install --namespace datascience stable/tensorflow-serving
 helm repo add 3loc https://3loc.github.io/charts helm repo update helm install mlflow 3loc/mlflow
```
