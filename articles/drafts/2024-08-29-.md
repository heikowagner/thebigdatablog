Detailed Explanation- Run the Container as Root:\
– By default, Docker containers run as the root user unless specified otherwise. You can start the container as root without any special options.

docker run -v /path/on/host:/path/in/container myimage

- Set Correct Permissions on the Mounted Volume:
  - When you mount a volume from the host to the container, you need to ensure that the permissions are set correctly so that files created by the root user inside the container are owned by the host user.
- Use –userns=host Option:
  - The –userns=host option ensures that the user namespace of the container matches the host. This means that the root user inside the container maps to the root user on the host, and similarly for other users.\
    ExampleLet’s go through an example step-by-step:
- Create a Dockerfile:
  - Create a simple Dockerfile that runs a command as root.

# Dockerfile

FROM ubuntu:latest\
RUN apt-get update && apt-get install -y sudo\
CMD [“bash”, “-c”, “echo ‘Hello from container’ > /path/in/container/hello.txt”]

- Build the Docker Image:
  - Build the Docker image from the Dockerfile.

docker build -t myimage .

- Run the Container with –userns=host:
  - Run the container with the –userns=host option to ensure the user namespace matches the host.

docker run –userns=host -v /path/on/host:/path/in/container myimage

- Verify File Ownership:
  - After the container runs, check the ownership of the file created inside the container.

ls -l /path/on/host/hello.txt

You should see that the file hello.txt is owned by the host user, even though it was created by the root user inside the container.\
Explanation of –userns=hostThe –userns=host option tells Docker to use the same user namespace as the host. This means that the root user inside the container is the same as the root user on the host. When you create files inside the container, they will have the same ownership as if they were created by the root user on the host.\
SummaryBy using the –userns=host option, you can run as the root user inside the container while ensuring that any files created are owned by the host user. This approach avoids the need for additional scripts or manual chown commands.\
